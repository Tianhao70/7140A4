The focus of this section of the Style Guide is ensuring user needs for usability and function are supported. We detail how the design facilitates effective and pleasurable interaction, aligning with behavioural design principles and accessibility standards. This involves demonstrating key interactive components and explaining the design rationale behind them.

Links

This section details the styling and behaviour of hyperlinks across the "Global Roots Community" website, ensuring they are discoverable, provide clear feedback, and meet accessibility requirements (WCAG 2.4.4 Link Purpose).

Annotation: Links are designed for clear discoverability and interaction feedback, adhering to Norman's principles. The default state uses a distinct color (var(--color-primary)) as a primary Signifier. The hover state introduces an underline and color change, providing immediate visual Feedback that the element is interactive. The focus state incorporates a visible outline (WCAG 2.4.7 Focus Visible), crucial for keyboard navigation accessibility. The active state provides momentary Feedback during activation. Link text must be descriptive to meet WCAG 2.4.4 (Link Purpose).

Menus & Navigation

Defines the structure and interaction patterns for the site's navigation systems, aiming to help users understand their location, explore effectively, and find information easily, adhering to principles like Consistent Navigation (WCAG 3.2.3).

Annotation: The proposed navigation aims for clarity and ease of access. For mobile (see mock-up below), a bottom navigation bar provides immediate access to core actions (Menu, Create, Share, Search), enhancing Discoverability. The hamburger menu icon acts as a Signifier for additional options, revealed in an overlay ('Menu expansion' mock-up). This overlay groups related functions logically (e.g., Account, Settings, Accessibility), supporting the user's Conceptual Model. Desktop navigation (prototyped in `components.css` via `#nav-desktop` for this style guide site, final design may vary) uses a consistent top bar. Active states (`aria-current="page"`) provide Feedback on location. Consistent placement and behaviour across the final site will adhere to WCAG 3.2.3. Accessibility for the proposed mobile menu requires the hamburger button to have `aria-expanded` state management and appropriate labels for icon-only buttons.

Cards

Cards are used to group related content snippets (like community topics/posts or replies) into digestible, self-contained units. This section shows the standard card layouts and interaction.

Annotation: Cards use consistent layouts to aid scannability and support the user's Conceptual Model (e.g., post previews include author, title, preview, and action). Clear visual hierarchy within cards prevents information overload (Constraints). Specific elements like the "View" button serve as clear Signifiers for action. Hover/focus states (to be defined in CSS) should provide Feedback. Semantic HTML structure (e.g., using `

Icons

Icons are used sparingly to reinforce meaning and guide actions. This section defines the icon style and usage guidelines, ensuring clarity and accessibility (WCAG 1.1.1 Non-text Content).

Annotation: A consistent icon set enhances usability and brand identity (Consistency). We aim for clear, universally understood symbols where possible (Signifiers, Mapping). Icons used for actions are typically paired with text labels for clarity. For icon-only buttons (like 'Edit' or 'Log out' shown below), an `aria-label` is crucial for screen reader users (WCAG 1.1.1). Recognizable brand icons (Google, Apple, Facebook) are used for third-party logins to leverage existing user familiarity (Conceptual Model).
Note: A final implementation would ideally use a single, consistent icon library (e.g., SVG icons or an icon font) for visual coherence and performance. These examples demonstrate usage principles.

Buttons

Details the design and interaction states for primary and secondary call-to-action buttons, ensuring clear affordance and feedback while meeting accessibility contrast requirements (WCAG 1.4.1 Use of Color, 1.4.11 Non-text Contrast).

Annotation: Buttons have clear visual cues (background color, padding) indicating clickability (Affordance/Signifier). Hover, focus, active (pressed), and disabled states (defined in CSS) provide distinct Feedback and manage Constraints. Labels clearly describe the action (e.g., "Submit", "Create Post"). Sufficient color contrast is maintained (WCAG 1.4.3, 1.4.11). Keyboard focus styles ensure visibility (WCAG 2.4.7).
