Tianhao's Work Phase 2

Accessibility Assessment

1. Introduction & Commitment

This assessment outlines the commitment and planned approach to ensuring the Connected Communities Initiative (CCI) website is accessible to all users, including those with disabilities. Accessibility is a core requirement outlined in the Client Brief and aligns with CCI's mission of inclusivity.

The target standard for this project is the Web Content Accessibility Guidelines (WCAG) 2.2 Level AA. Meeting this standard ensures a robust, usable, and equitable experience for diverse audiences, including international students, migrants, and local Australians engaging with the platform.

2. Assessment Methodology

The accessibility assessment will be an ongoing process throughout the design and development phases (Work Phase 3 & 4), employing a combination of methods:

  • Guideline Adherence: Directly referencing the WCAG 2.2 Quick Reference Guide to understand and implement specific Success Criteria (SC) for Level AA.
  • Framework Application: Structuring the assessment around the four core principles of WCAG: Perceivable, Operable, Understandable, and Robust (POUR).
  • Manual Testing:
    • Keyboard Navigation: Ensuring all interactive elements (links, buttons, form controls, custom components) are reachable and operable using only the keyboard, with a clear visible focus indicator.
    • Screen Reader Simulation: Using tools like NVDA, VoiceOver, or JAWS to check content structure, semantic meaning, alternative text for images, and ARIA announcements.
    • Zoom & Reflow: Testing page layout and readability when zoomed up to 200%.
    • Contrast Check: Verifying text and UI elements meet required contrast ratios.
  • Automated Tools (Auxiliary): Potentially using browser extensions (e.g., WAVE, Axe DevTools) or online checkers for initial scans and identifying common issues, understanding these tools are supplementary to manual checks.
  • Code Review: Ensuring the use of semantic HTML, correct ARIA implementation where necessary, and valid code.

3. Assessment Based on WCAG 2.2 AA Principles (POUR)

The following outlines key considerations under each WCAG principle relevant to this project:

Perceivable

Information and user interface components must be presentable to users in ways they can perceive.

  • 1.1 Text Alternatives: All meaningful images, icons, and other non-text content will have appropriate text alternatives (e.g., alt attributes for images). Decorative elements will be marked appropriately (alt="") to be ignored by assistive technology. (WCAG 1.1.1 - Level A)
  • 1.3 Adaptable: Content will be structured using semantic HTML (<nav>, <main>, <h1>-<h6>, <ul>, etc.) to ensure a logical reading order and structure understandable by assistive technologies. (WCAG 1.3.1, 1.3.2 - Level A)
  • 1.4 Distinguishable:
    • Color will not be used as the sole means of conveying information (e.g., links will have visual indicators beyond color). (WCAG 1.4.1 - Level A)
    • Text and background colors will meet the AA contrast ratio of 4.5:1 for normal text and 3:1 for large text. (WCAG 1.4.3 - Level AA)
    • Text will be resizable up to 200% without loss of content or functionality, using relative units (rem, em) for text sizing. (WCAG 1.4.4 - Level AA)
    • Content reflow will be managed for smaller viewports and zoomed views. (WCAG 1.4.10 - Level AA)
    • Sufficient contrast for non-text elements (UI components, graphics) will be ensured. (WCAG 1.4.11 - Level AA)

Operable

User interface components and navigation must be operable.

  • 2.1 Keyboard Accessible: All functionality will be operable via a keyboard interface. No keyboard traps will exist. (WCAG 2.1.1, 2.1.2 - Level A)
  • 2.4 Navigable:
    • A mechanism to bypass blocks of content (e.g., "Skip to main content" link) will be considered. (WCAG 2.4.1 - Level A)
    • Page titles (<title>) will be clear and descriptive. (WCAG 2.4.2 - Level A)
    • Focus order will be logical and preserve meaning. (WCAG 2.4.3 - Level A)
    • The purpose of links will be clear from the link text alone or programmatically determined context. (WCAG 2.4.4 - Level A)
    • Multiple ways to navigate the site (e.g., navigation menu, site map, search if implemented) will be considered. (WCAG 2.4.5 - Level AA)
    • Headings and labels will describe topic or purpose. (WCAG 2.4.6 - Level AA)
    • Keyboard focus will be clearly visible and discernible. Default focus indicators will not be removed without a clear replacement. (WCAG 2.4.7 - Level AA)
    • Focus appearance will have sufficient contrast and thickness. (WCAG 2.4.11 - Level AA)
    • Any focus not obscured by other content will be ensured. (WCAG 2.4.12, 2.4.13 - Level AA)
  • 2.5 Input Modalities: Ensure functionality can be operated via various inputs beyond keyboard (relevant for potential PWA features). Target sizes for pointer inputs will be sufficient. (WCAG 2.5.5, 2.5.8 - Level AA)

Understandable

Information and the operation of user interface must be understandable.

  • 3.1 Readable:
    • The primary language of the page (English) will be programmatically set (<html lang="en">).
    • Content will be written as clearly and simply as possible, avoiding jargon where feasible, aligning with user needs for simple English. (WCAG 3.1.5 - Level AAA, aiming for best practice here).
    • Mechanisms for identifying definitions of unusual words or abbreviations will be considered if necessary. (WCAG 3.1.3, 3.1.4 - Level AAA)
  • 3.2 Predictable:
    • Navigation menus will be consistent across pages. (WCAG 3.2.3 - Level AA)
    • UI components used repeatedly will be identified consistently. (WCAG 3.2.4 - Level AA)
  • 3.3 Input Assistance:
    • Errors in forms (like the newsletter signup or potential dynamic feature inputs) will be clearly identified, described in text, and suggestions for correction provided where possible. (WCAG 3.3.1 - Level A)
    • Labels and instructions will be provided for user inputs. (WCAG 3.3.2 - Level A)
    • Error prevention mechanisms (e.g., confirming submissions for critical actions) will be implemented where appropriate. (WCAG 3.3.4 - Level AA for critical errors)
    • Redundant entry will be minimized where feasible. (WCAG 3.3.7 - Level A)

Robust

Content must be robust enough that it can be interpreted reliably by a wide variety of user agents, including assistive technologies.

  • 4.1 Compatible:
    • Code (HTML, CSS) will validate against specifications where possible.
    • Elements will have complete start/end tags, be nested correctly, and avoid duplicate IDs.
    • ARIA attributes will be used correctly to enhance accessibility for custom controls or dynamic content updates, supplementing (not replacing) native HTML semantics where appropriate. Status messages providing feedback without interrupting the user will use appropriate techniques (e.g. ARIA live regions). (WCAG 4.1.1, 4.1.2 - Level A, 4.1.3 - Level AA)

4. Specific Component Considerations

Key components from the current design will be specifically reviewed:

  • Navigation Menu (`.navbar`): Ensure full keyboard operability (toggle, links), visible focus states, correct use of `aria-current="page"`, and potentially `aria-expanded` on the toggle button managed via JavaScript (`js/index.js` likely).
  • Flip Cards (`.flip-card`):** Check keyboard activation (currently uses `:focus-visible`), ensure content is accessible to screen readers in both states, verify meaningful `aria-label` for context.
  • Interactive Highlights (`.interactive-highlight`, `.interactive-highlight-stories`):** Confirm keyboard focus (`tabindex="0"`). Assess how the tooltip information (currently CSS `::after`) is conveyed to assistive technologies. If `::after` content is not reliably read, alternative techniques (like dynamically shown/hidden elements linked with `aria-describedby`) will be explored.
  • Interview Transcripts (`.interview` structure): Ensure logical heading structure (`h5`, `p.question`, `.response`) and list formatting within responses are correctly interpreted by screen readers.
